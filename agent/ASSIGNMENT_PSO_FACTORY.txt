# Assignment: Vulkan PSO factory and cache

Goal: Introduce a standalone, self-contained PSO (Pipeline State Object) factory that builds Vulkan graphics pipelines on demand and caches them to avoid redundancy. Build on top of rapid_vulkan's CommandBuffer, GraphicsPipeline, Drawable, DrawPack and other utility classes; modify the command buffer manager and the rapid_vulkan library as needed to support this. The factory must support efficient lookup of existing PSOs via a compact key structure (bit fields for memory efficiency). The generic draw path must support many draw variants (no vertex input vs per-vertex format/stride, different shaders, etc.). The factory lives as a data member of GpuContextVulkan. When the PSO factory is complete, redo the generic draw action (GpuDrawVulkan) to use the factory and the rapid_vulkan recording path (Drawable → DrawPack → CommandBuffer::render).

Key design (memory-efficient, fast lookup)
---

The PSO key must enable efficient lookup: small size (good cache behavior, fast hashing) and unambiguous equality. Use bit fields so the key is as compact as possible.

- Shader identity: Use a 64-bit hash derived from (vertex module, fragment module, entry point names) so the key does not store pointers; same shader combo always produces the same hash.
- Vertex input: Use bit fields — e.g. no_vertex_input:1; when present: vertex_stride (e.g. 16 bits, max 64K); and a 32-bit hash of GpuGeometry::VertexFormat (attributes) so different formats get different keys without storing variable-length data.
- Color format: vk::Format fits in 16 bits; use a bit field.
- Layout: Push constant size (e.g. 8 bits, 0..256) and optionally descriptor set layout count (2 bits) as bit fields; or a 32-bit layout key if layout cache is separate.
- Fixed state (topology, rasterization, blend, multisample, dynamic state) can be omitted and applied in the builder for current draws.
- Total key size: aim for one or two 64-bit words (8–16 bytes) so the key is cheap to copy and hash. Implement operator== and std::hash<GraphicsPsoKey> for O(1) lookup in std::unordered_map.

PHASE 1: PSO descriptor and key types (bit-field key)
---

Goal: Define the immutable key that uniquely identifies a graphics PSO for caching. The key must use bit fields for memory efficiency and be hashable and comparable for efficient lookup (see Key design section below).

[ ] Task 1.1 — Add GraphicsPsoKey with bit fields for memory-efficient key
    - Introduce GraphicsPsoKey (in vk-pso-factory.h) that encodes everything needed to build one graphics PSO and to detect duplicates, using bit fields to minimize size:
      - Shader: 64-bit hash of (vertex module handle, fragment module handle, vs entry name, ps entry name) so the key is self-contained and pointer-free.
      - Vertex input: bit fields — no_vertex_input:1; vertex_stride:16 (0 when no input); vertex_format_hash:32 (hash of GpuGeometry::VertexFormat when has input).
      - Color format: bit field (e.g. 16 bits) storing vk::Format for the single color attachment.
      - Layout: bit fields for push_constant_size:8 (e.g. 0..256); reserved or layout_id for future descriptor set layout.
    - Keep total key size to one or two 64-bit words (8–16 bytes). Implement operator== and std::hash<GraphicsPsoKey> for efficient map lookup.
    - Fixed state (topology eTriangleList, rasterization, blend, multisample, dynamic viewport/scissor) is applied in the builder, not in the key.
    - Verification: Build passes. Key type compiles; unit test that two keys with same field values are equal and have same hash; two keys differing in one field are unequal and (with high probability) different hash. Commit with message e.g. "Assignment PSO factory: add GraphicsPsoKey with bit fields (Task 1.1)".

[ ] Task 1.2 — Add pipeline layout cache / get-or-create layout
    - Pipeline layout is shared when push constant range and descriptor set layout(s) match. Either:
      (A) Cache pipeline layouts in the factory by a small “LayoutKey” (push range + set layout handles), or
      (B) Have the factory create layout on demand per PSO key and cache only the full PSO (layout owned by each cached entry).
    - Recommended: (A) Separate layout cache so multiple PSOs can share one layout. Define LayoutKey (push range, 0..N set layout handles), implement getOrCreateLayout(LayoutKey) -> vk::PipelineLayout; store layouts in a map, destroy in factory destructor.
    - Verification: Build passes. Factory (or a minimal stub) can return a layout for a given LayoutKey; second call with same key returns same layout. Commit with message e.g. "Assignment PSO factory: add pipeline layout cache (Task 1.2)".
