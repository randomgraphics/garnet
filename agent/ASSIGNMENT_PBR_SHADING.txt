--- PLAN: SharedShaderConstants (public interface and behavior) ---

SharedShaderConstants will eventually provide a set of uniform buffers and textures containing shared GPU resources that are referenced or updated by individual effects (e.g. PbrShading).

Public interface (pipeline.h):
- Must NOT expose any information about how data is organized on the GPU. Layout, binding slots, descriptor set structure, and buffer/texture allocation are entirely the responsibility of the API-specific implementation (e.g. Vulkan backend in core/rdg).
- Must provide at least a set of setters so the caller can supply logical data. The caller passes high-level data (frame counter, view matrix, render target, lights, etc.); the implementation copies or references this data and decides how to pack it into uniform buffers and textures for the active graphics API.
- Suggested logical data and setters (to be added or refined in implementation):
  - FrameInformation: frameCounter, frameDuration (Microseconds). setFrameInformation(const FrameInformation &).
  - ViewInformation: full world→view→clip transformation (e.g. worldToView, viewToClip matrices), camera position in world space (Location / WorldUnit) so implementation can translate model and light locations to float3 at runtime, optional RenderTarget reference. setViewInformation(const ViewInformation &).
  - DirectLightingInformation: list of direct lights (type, position, direction, color, range, cone angles, etc.). setDirectLightingInformation(const DirectLightingInformation &).
- CreateParameters: at least AutoRef<GpuContext> gpu. create(db, name, params) returns an implementation that satisfies the above.
- No getters that return GPU layout, buffer pointers, or binding indices in the public header. Any such details live in the implementation and are used when building effect tasks (e.g. when PbrShading::build() fills GpuDraw::A from SharedShaderConstants).

Implementation (core/rdg, backend-specific):
- Backend implements the setters by storing the logical data and/or uploading to its own uniform buffers and textures. When an effect (e.g. PbrShading) builds a draw task, it obtains from SharedShaderConstants whatever is needed to fill GpuDraw::A (e.g. descriptor sets or resource refs), using backend-specific code paths only in core/rdg, not in the public API.

--- PLAN: Embedded resources (CMake) ---

To avoid runtime file loading, PBR assets (textures, material data, optionally mesh data) are embedded into the sample executable. Use CMake to drive platform-specific embedding:

- **Windows:** Use a .rc (resource script) file. In the .rc file, add RCDATA (or custom resource type) entries that reference the binary asset files (e.g. .png, .jpg, .bin). Add the .rc file to the sample target in CMake (e.g. add it to the executable's sources so the MSVC resource compiler compiles it and the linker links the resource section). At runtime, the sample uses the appropriate Win32 API (e.g. FindResource / LoadResource / LockResource) to get a pointer and size to the embedded blob, then wraps it in a GN::MemFile or similar and passes it to Material::load(db, name, {.gpu = gpu, .source = &memFile}).

- **Linux (and similar):** Use objcopy to turn each asset file into an object file that exposes symbols (e.g. _binary_foo_png_start, _binary_foo_png_end, _binary_foo_png_size). CMake: add_custom_command to run objcopy (e.g. objcopy --input-target=binary --output-target=elf64-x86-64 --rename-section .data=.rodata,contents,alloc,load,readonly,data foo.png foo.png.o). Add the generated .o files to the sample target so they are linked into the executable. At runtime, the sample uses the start/size symbols to wrap the data in GN::MemFile and pass to Material::load.

- **CMake:** Keep platform logic in the sample's CMakeLists (or a shared .cmake include): if(WIN32) add .rc and list asset files in the .rc; else() add_custom_command + target_sources for objcopy-generated .o files. Optionally support both embedded (default) and "load from disk" for development (e.g. a CMake option or runtime path override).

--- PLAN: SubGraph (self-contained workflows; submit and drop) ---

SubGraph is a self-contained list of workflows, together with their tasks and task arguments, that describe an effect (e.g. PBR) and can be submitted to a RenderGraph to render that effect to the screen or to a render target texture.

Behavior and contents:
- SubGraph holds: builtResult (Action::ExecutionResult from the effect’s build step) and a list of Workflow * (the workflows created by the effect’s build()). Each workflow already contains its tasks and arguments; no separate “task list” is needed at SubGraph level beyond the workflows.
- Typically produced by effect::build(params): the effect creates one or more workflows on a RenderGraph (from params), appends tasks (e.g. ClearRenderTarget, GpuDraw) with arguments, and returns SubGraph{PASSED, {workflow1, ...}}. The SubGraph thus fully describes what to execute.

Utility methods (to be implemented):
- submit(RenderGraph & graph, name): Submit this subgraph for execution. Implementation: call graph.submit(SubmitParameters{.workflows = this->workflows, .name = name}). After submit, the workflows are “in flight”; optionally clear the SubGraph’s workflow list to prevent double-submit. Return the Submission object from graph.submit(...).
- drop(RenderGraph & graph): Drop this subgraph without executing. Implementation: for each workflow in this SubGraph, call graph.dropWorkflow(workflow); then clear this SubGraph’s workflow list so the SubGraph is empty and no longer references those workflows.

Placement:
- Declarations in pipeline.h (SubGraph is part of the pipeline API). Definitions can live in a .cpp that is part of the rdg core (e.g. pipeline.cpp or subgraph.cpp) and linked into the same module that implements RenderGraph::submit and dropWorkflow, so that submit/drop call into RenderGraph without circular dependency.

Usage in assignment:
- PbrShading::build(params) returns a SubGraph. The sample (or any caller) can then either subgraph.submit(renderGraph, "PBR") to run it, or subgraph.drop(renderGraph) to discard it. This keeps the assignment consistent with “SubGraph is self-contained and can be submitted or dropped.”
